## 1. Core Law

> **Do not scale the loop before proving the slice.**

Every APEX module begins by building and verifying the smallest complete end-to-end pathway capable of demonstrating meaningful value, truth integrity, evidence capture and human control.

```text
DEFAULT OPERATION
Observe current state and target value
→ Select the smallest proof-bearing vertical slice
→ Build the complete slice end-to-end
→ Test behavior and truth boundaries
→ Capture evidence, failures, cost and benefit
→ Decide whether the loop is worth repeating or scaling
→ Only then consider APEX 1M activation
```

## 2. Definition: Verified Vertical Slice

A **Verified Vertical Slice** is the smallest bounded implementation that crosses all essential layers required to prove a useful loop rather than merely producing a design, isolated component or attractive prototype.

A qualifying vertical slice must contain:

| Required Element | Meaning |
|---|---|
| Purpose / Benefit Target | The value being tested is explicitly declared. |
| Real End-to-End Path | Input → process → output → evidence → decision can be exercised. |
| Bounded Scope | The slice is small enough to test and correct without uncontrolled cost or risk. |
| Acceptance Criteria | Completion and failure conditions are falsifiable. |
| Truth Labels | Implemented, tested, proposed, blocked and aspirational capability are separated. |
| Evidence Receipt | Results, failures, limitations and version/repository route are recorded. |
| Human Control | Promotion, stopping, scaling or deployment remains reviewable and reversible where applicable. |

## 3. Default Execution Rule Across All Modules

| Module Condition | Required Operation |
|---|---|
| New module or major new capability | Begin with a vertical-slice definition, not broad-scale iteration. |
| Strong concept but no working artifact | Build the smallest executable/inspectable path first. |
| Visual prototype exists but proof is absent | Test a bounded behavior path and evidence it before scaling. |
| Evidence-bearing slice passes | Compare benefit, risk, cost, repeatability and next scale level. |
| Slice fails or benefit is weak | Repair, pivot, hold or stop; do not activate mass scaling. |
| Scope changes materially | Re-run vertical-slice validation for the changed loop. |

## 4. Conditional Activation Rule: APEX 1M / 🧪Ⓥ1M🅾️♾️💯

`APEX 1M / 🧪Ⓥ1M🅾️♾️💯` represents **scaled iteration/testing mode**. It is not the default entry point and is not automatically triggered by enthusiasm, conceptual elegance, mockup quality or proxy convergence alone.

### Activation Requires All Gates

| Gate ID | Activation Gate | Required Proof |
|---|---|---|
| `VVSF-G01` | Vertical slice complete | A working bounded end-to-end artifact or process run exists. |
| `VVSF-G02` | Acceptance tests passed or responsibly bounded | Test results and any known failures are recorded. |
| `VVSF-G03` | Loop value established | Evidence indicates that repeating/scaling the loop could produce meaningful benefit. |
| `VVSF-G04` | Scale target defined | What will be scaled, how far, with what stop rules and what metric is stated. |
| `VVSF-G05` | Risk and cost bounded | Known risks, compute/time/cost burden and external dependencies are assessed. |
| `VVSF-G06` | Evidence route active | Repository/evidence-ledger record path is established. |
| `VVSF-G07` | Human-control gate present | Operator may approve, hold, reduce, switch, stop or roll back the scale action. |
| `VVSF-G08` | No unsupported promotion | The slice is not represented as real-world proof beyond the measured scope. |

### Activation Decision

```text
IF all VVSF gates are satisfied
AND benefit of scaling > cost/risk/complexity burden
AND human control remains visible
THEN status = SCALE_CANDIDATE_FOR_APEX_1M_REVIEW
ELSE status = REMAIN_IN_VERTICAL_SLICE_OR_REPAIR_MODE
```

Activation requires an explicit scale decision. `APEX 1M` does not by itself authorize one million executions, uncontrolled automation, large external writes, production deployment or unreviewed cost consumption.

## 5. Scale Levels After Slice Verification

Scale should be earned progressively rather than jumping immediately to maximum volume.

| Scale State | Meaning | Use Condition |
|---|---|---|
| `VS-0` | Slice defined, not run. | Initial specification. |
| `VS-1` | One bounded slice run completed. | Initial behavior/evidence test. |
| `VS-R` | Slice under repair/retest. | Failure or weak proof found. |
| `VS-P` | Slice proven within declared scope. | Acceptance criteria met with evidence. |
| `SCALE-10` | Small repeated-loop check. | Test repeatability and failure patterns. |
| `SCALE-100` | Broader bounded validation. | Value and stability confirmed at small scale. |
| `APEX-1M-CANDIDATE` | Mass-scale plan proposed. | All activation gates complete; operator review required. |
| `APEX-1M-ACTIVE` | Approved scoped scale execution. | Approved volume, cost, safety, logging and stop rules exist. |
| `HOLD / STOP / ROLLBACK` | Scaling denied or reversed. | Benefit insufficient, drift/failure/risk detected or control lost. |

## 6. Relationship to C∞ and Proxy MSE

C∞ convergence remains a bounded build-improvement mechanism, but it must operate under this policy:

```text
Vertical Slice proves what loop is valuable.
C∞ helps improve that proven bounded loop.
APEX 1M scales only the loop worth scaling.
```

A low proxy MSE, convergence score or internal test residual is not proof of real-world benefit, transfer, safety, adoption or production readiness unless those outcomes are directly measured within scope.

## 8. Required Iteration Automater Behavior

The Automater must:

1. default every new active workstream to `VERIFIED_VERTICAL_SLICE_FIRST`;
2. reject or hold scale requests without a linked vertical-slice evidence receipt;
3. rank the smallest complete proof-bearing action above broad unproven architecture expansion;
4. monitor whether iteration is producing verified benefit rather than merely more output;
5. surface `STAY`, `SWITCH`, `HOLD`, `STOP`, `ROLLBACK` and `SCALE` choices through APEX Mirror;
6. generate a scale-candidate packet only after proof gates are met;
7. preserve every decision in the Evidence Ledger.

## 9. Stop / Hold / Rollback Conditions

| Condition | Required Response |
|---|---|
| No end-to-end slice defined | `BLOCKED — DEFINE VERTICAL SLICE` |
| Slice lacks falsifiable proof criteria | `HOLD — ADD ACCEPTANCE CRITERIA` |
| Evidence missing or unlabeled | `HOLD — EVIDENCE REPAIR REQUIRED` |
| Failure rate or risk materially increases during scale | `STOP OR ROLLBACK — INVESTIGATE FAILURE` |
| Scaling produces volume without meaningful benefit | `STOP — NO-GAIN RULE` |
| Scale weakens human visibility/control | `BLOCK — HUMAN CONTROL SUPREMACY VIOLATION` |
| Proxy metric improves while real outcome worsens or is unknown | `HOLD — METRIC-FIT REVIEW` |
